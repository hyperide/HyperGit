# Treat the SwiftUI mobile MVP as the current product role

## 1. Problem Frame

The app needs a clear role in the larger HyperGit system so agents can make
provider, auth, cache, and UI choices without reopening platform scope.

## 2. Decision

**Selected:** The current role of HyperGit in this repository slice is the SwiftUI mobile MVP for browsing GitHub and Linear data locally.

**Selection policy:** Use `docs/SPEC.md` section 2 and the user's current-focus instruction as the governing role for this continuation.

**Why selected:** It anchors implementation to the mobile app's provider clients, stores, OAuth/token handling, and local-first behavior.

**Spec sections:**
- TS.hypergit.mobile.001

## 3. Rationale

**Counterargument:** HyperGit's long-term role is a full GitHub replacement, not just a mobile viewer.

**Selected variant weakest link:** The role must be refreshed when backend work becomes active or when the app starts depending on HyperGit-specific server APIs.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Treat all HyperGit platform roles as active | Rejected | It would make app work depend on unstarted server commitments. |
| Treat the app as GitHub-only | Rejected | Linear and future HyperGit provider abstraction are explicit parts of the spec. |

**Evidence requirements:**
- `swift build`
- `swift test`
- GitHub and Linear client tests

## 4. Consequences

**Rollback plan:**
Triggers:
- The current user focus changes from app work to backend/platform work.
- Provider abstractions no longer cover the mobile MVP role.

**Refresh triggers:**
- New providers are added, HyperGit backend support starts, or the mobile MVP definition changes.
