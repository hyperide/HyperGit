# Make offline mobile access the active target outcome

## 1. Problem Frame

The continuation needs a target-system statement that says what changes for the
user now, before the future HyperGit backend exists.

## 2. Decision

**Selected:** The active target outcome is offline-readable SwiftUI mobile access to GitHub and Linear data after sync.

**Selection policy:** Prefer the concrete mobile outcome already stated in `docs/SPEC.md` section 2 and reiterated by the user over broader platform goals.

**Why selected:** It makes the app's observable purpose explicit and keeps implementation work tied to repository, pull request, issue, commit, file, and ticket access.

**Spec sections:**
- TS.hypergit.mobile.000

## 3. Rationale

**Counterargument:** The platform vision is broader than mobile offline browsing, so this target can look too narrow.

**Selected variant weakest link:** The statement depends on sync and cached data paths continuing to work as the app adds providers.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Treat the backend as the active target | Rejected | The current implementation focus is the app, not server/platform work. |
| Leave the target implicit in `docs/SPEC.md` only | Rejected | Haft onboarding needs a concrete active target section for local governance. |

**Evidence requirements:**
- `swift build`
- `swift test`
- AppStore local-first fallback tests

## 4. Consequences

**Rollback plan:**
Triggers:
- The user explicitly changes the active focus away from the app.
- Mobile cache/offline behavior is removed from the product spec.

**Refresh triggers:**
- Mobile scope expands beyond GitHub and Linear, or HyperGit backend support becomes the active implementation phase.
